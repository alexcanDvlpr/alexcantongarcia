Lo más normal actualmente, cuando creamos un software o entramos en un equipo de desarrollo, es que tengamos el código de nuestro proyecto centralizado y controlado, es decir, saber qué cambios se han hecho, cuándo y por quién. Principalmente porque si desarrollamos el típico Hola Mundo no tiene mucho sentido, pero si estas en un proyecto amplio y con más compañeros puede ser de lo mas socorrido.

Para poder tener nuestro código salvaje bajo control, podemos usar un **Control de Versiones**

## ¿Qué es un Control de Versiones?

Como su nombre indica, es un sistema para tener control sobre las distintas versiones que se puedan tener de nuestro software. Quizá queremos ver el primer cambio (commit) que hicimos y deshacerlo, pues con un control de versiones es posible, uno de los más conocidos y potentes es **Git**.

## ¿Qué es Git?

Git es a día de hoy, el sistema de control de versiones más conocido y usado del mundo, creado por **Linus Torvals**, desarrollador del Kernel de Linux. Su principal característica es que es **distribuido**, esto quiere decir que múltiples personas pueden trabajar en equipos sobre un mismo proyecto.

En este post, te voy a dejar algunos de los comandos más básicos y que necesitas saber para empezar a trabajar con Git.

## ¿Qué es GitHub?

GitHub es un servicio web, que nos permite a los desarrolladores almacenar nuestro código en la nube, y poder acceder a él a través de la plataforma web. Tambien nos permite tener creado un perfil para gestionar nuestros proyectos o repositorios y usarlo así a modo de red social.

Es propiedad de **Microsoft** y nos permite tener repositorios públicos o privados. Por cada repositorio nos permite tener una sección de **issues**, **integración contínua** y **entrega contínua**, lo que se denomina **CI/CD**.

Puedes visitar GitHub aquí: [https://github.com/](https://github.com/)

## ¿Cómo empezar?

Como he dicho, se puede usar Git con más compañeros de equipo, esto quiere decir, que tendremos que tener nuestro código en la nube. Para esto existen distintos proveedores de repositorios de código, que no dejan de ser almacenes de código creados por organizaciones o por nosotros mismos.

Posiblemente todo lo que te he contado hasta ahora ya lo sepas, y solo quieres aprender los comandos básicos, pues no te voy a hacer esperar más.

### git clone

Con este comando nos descargaremos un proyecto que ya resida en uno de los proveedores anteriormente comentados, para descargarlo en local simplemente necesitaremos la URL del proyecto, disponible en el repositorio, por ejemplo de GitHub:

![URL Clone](/images/posts/empezar-con-git-en-2021/clone.webp)

En este caso, copiamos la URL y ejecutamos en una consola:

```shell
git clone <url_proyecto>
```

Si por cualquier razón, no queremos que el proyecto en local se llame igual que en remoto, podemos añadirle un alias de la siguiente manera:

```shell
git clone <url_proyecto> <alias_proyecto>
```

### git init y git remote

Sin embargo, si lo que queremos hacer es crear un proyecto totalmente nuevo y gestionarlo con git, tendremos que acceder a la raíz de nuestro proyecto e inicializar git, para ello ejecutaremos el siguiente comando:

```shell
git init
```

Una vez ejecutado, se generará un directorio en la raíz llamado `.git`, en este directorio se almacenará toda la información, como pueden ser las conexiones remotas, las ramas locales etc.

En cambio, iniciar un proyecto de git, solo nos sirve para operar en local, si queremos subir nuestro código a un repositorio en remoto hay que conectar nuestro repositorio local con el remoto de la siguiente manera:

```shell
git remote add <origen> <url_proyecto>
```

Normalmente por nomenclatura, el origen del repositorio suele ser **origin** (de origen), pero puedes tener más conexiones remotas, como por ejemplo origin para el desarrollo en local y otro repositorio remoto que sea del cliente que le llames delivery.

### git add

Cuando en nuestro proyecto hemos hecho cambios o añadido nuevos ficheros, y los queremos añadir a la fase de preparación de git, tendremos que usar el comando `add` con un parametro, nombre o nombres de los ficheros que queremos añadir o un `.` para añadir todo.

```shell
# Por ficheros
git add archivo1.txt test/test1.js

# Añadir todo
git add . 
```

Este paso nos permite poder crear un commit (un mensaje descriptivo) a esos cambios.

### git commit

Este comando nos ayuda a añadir un mensaje descriptivo de los cambios realizados y añadidos con el comando `git add`. Los mensajes de commit, dan para otro artículo extenso, pero un buen commit tiene que explicar de forma breve y concisa qué hacen esos cambios que se han realizado, por pocos o muchos que sean. 

Si ejecutamos directamente `git commit` se nos abrirá el editor de texto que tengamos configurado por defecto, como pueden ser **nano**, **vi**, **vim** o **VSCode** y ahí podremos escribir nuestro commit, sin embargo, si el commit va a ser un mensaje relativamente corto, podemos usar el flag `-m`:

```shell
git commit -m "Aqui iría nuestro mensaje descriptivo de nuestros cambios"
```

Con esto ya tendríamos los cambios preparados y descritos, el siguiente paso sería subirlo a nuestro repositorio.

### git push

El comando `git push` es el que usaremos para poder subir nuestros cambios ya preparados a nuestro repositorio remoto, previamente configurado con `git remote`

Una vez tengamos los cambios preparados y descritos correctamente con un commit, ejecutaremos:

```shell
git push <nombre_origen> <nombre_rama>
```

Y esto subirá los cambios a nuestro repositorio remoto, al origen y rama (por defecto **master**) que le indiquemos, como podría ser lo siguiente:

```shell
git push origin master
```




### Conclusión
Git es un sistema de control de versiones tan amplio como complejo, pero empezar es tan simple como conocer estos comandos. Con estos comandos ya puedes empezar a familiarizarte con el sistema de git, la consola de comandos, con algún proveedor como **GitHub** y una vez dominemos esto, seguir ampliando conocimientos dentro de este sistema de control de versiones.
